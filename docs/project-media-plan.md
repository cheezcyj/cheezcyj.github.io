# 프로젝트 미디어 준비 계획

작성일: 2026-07-16
대상 프로젝트: `CHEEZCYJ Portfolio Redesign`
현재 상태: 실제 cover와 gallery 미승인, CSS 기반 neutral fallback 사용

## 1. 기본 규격

- cover 비율은 카드와 상세 hero 모두 `16:9`를 사용한다.
- 권장 원본 크기는 `2400 × 1350px`이며, 최소 `1600 × 900px` 이상을 준비한다.
- 원본은 브라우저 UI와 텍스트가 선명하게 보이는 PNG로 보관하고, 공개본은 화질을 확인한 WebP 또는 AVIF를 우선 검토한다.
- 카드와 상세 화면이 같은 비율을 사용하므로 별도 crop 없이 재사용하되, 핵심 UI는 화면 중앙 80% 안에 배치한다.
- 실제 cover가 승인되기 전에는 현재 token 기반 media fallback을 유지한다.

## 2. 화면별 주요 영역

### 홈 카드

- 작은 크기에서도 사이트의 다크 네이비, 크림과 골드 시각 언어가 식별되어야 한다.
- 프로젝트 섹션 제목, 카드 또는 주요 Hero 중 하나를 명확한 초점으로 삼는다.
- 좌우와 상하 약 10%는 카드 축소와 다양한 화면 밀도를 위한 안전 영역으로 비운다.
- 작은 본문이나 코드처럼 축소 시 읽히지 않는 정보를 주요 초점으로 사용하지 않는다.

### 상세 hero

- Astro 결과물의 전체 레이아웃과 브랜드 분위기를 먼저 전달한다.
- 헤더, Hero와 프로젝트 카드처럼 프로젝트 성격을 설명하는 실제 UI를 우선한다.
- 브라우저 탭, 주소창, 개발 도구와 운영체제 UI는 의도적인 설명이 아니면 제외한다.

## 3. 추천 캡처 후보

### 데스크톱

1. 메인 Hero와 Development Projects 섹션이 함께 보이는 화면
2. 프로젝트 카드와 수평 레일의 카드 폭·여백이 드러나는 화면
3. 프로젝트 상세 페이지 상단의 제목, meta와 media 구성
4. 콘텐츠 공개 정책이나 자산 검증 구조를 시각적으로 설명할 수 있는 실제 사이트 화면

### 모바일

1. 모바일 메인 Hero
2. 전체 화면 메뉴의 열린 상태와 키보드 접근 가능한 내비게이션
3. 다음 카드가 이어질 수 있는 홈 프로젝트 레일
4. 프로젝트 상세 페이지의 1열 제목, fallback 또는 승인 cover와 meta

코드 편집기나 터미널 화면은 실제 디자인 화면을 보조할 때만 사용하며 대표 이미지보다 우선하지 않는다.

## 4. Gallery 권장 구성

- 권장 이미지 수는 `4~6개`다.
- 같은 화면의 미세한 변형을 반복하지 않고 데스크톱, 모바일, 내비게이션, 카드/레일, 상세 화면과 콘텐츠 구조를 고르게 포함한다.
- 각 이미지는 하나의 구현 결과나 설계 판단을 설명할 수 있어야 한다.
- gallery 순서는 전체 인상 → 핵심 UI → 반응형/접근성 → 구조 설명 순을 권장한다.

## 5. 파일명과 위치

권장 파일명 형식:

```text
cheezcyj-portfolio-redesign-{view}-{viewport}.{ext}
```

예시:

```text
cheezcyj-portfolio-redesign-home-desktop.webp
cheezcyj-portfolio-redesign-menu-mobile.webp
cheezcyj-portfolio-redesign-project-detail-desktop.webp
```

- 영문 소문자, 숫자와 하이픈만 사용한다.
- `final`, `new`, `screenshot1`처럼 의미가 불분명한 이름을 피한다.
- 현재 content schema는 root-relative `cover.src`를 사용하므로 1차 권장 위치는 `public/images/projects/cheezcyj-portfolio-redesign/`이다.
- 이후 Astro Image와 `src/assets` 기반 이미지 참조를 도입하려면 schema와 legacy URL 정책을 함께 변경하고 별도 검증한다.
- 편집용 원본과 공개 최적화본을 구분해 관리한다.

## 6. 한국어 alt 작성 기준

- 이미지가 보여 주는 화면과 핵심 목적을 한 문장으로 설명한다.
- `이미지`, `스크린샷`, `사진`처럼 이미 알려진 형식부터 반복하지 않는다.
- 색상만 나열하지 않고 화면 구조나 기능을 함께 설명한다.
- 카드와 본문에서 같은 이미지가 같은 의미로 반복되면 동일한 핵심 설명을 유지한다.
- 장식 목적이 아닌 실제 프로젝트 화면이므로 빈 alt를 사용하지 않는다.

예시:

> 다크 네이비 배경 위에 프로젝트 수평 레일과 골드 강조를 배치한 포트폴리오 메인 화면

## 7. 공개 전 점검

- 이메일, 토큰, 로컬 경로, 계정명 등 민감 정보 노출 여부
- 브라우저 북마크, 알림, 다른 탭과 운영체제 사용자 정보 노출 여부
- 개발 도구, 콘솔 오류와 디버그 overlay 노출 여부
- 임시 문구와 Draft Preview 표시가 최종 공개 이미지에 남아 있지 않은지
- 이미지 사용 권리와 공개 범위가 Owner에게 승인됐는지
- 실제 Astro 화면과 현재 구현 상태를 정확히 보여 주는지
- alt, width, height와 최적화 형식이 확정됐는지

## 8. 사용 금지 원칙

- v0의 design/frontend 샘플 자산을 실제 프로젝트 이미지로 사용하지 않는다.
- 기존 Jekyll의 cabin, cake 등 템플릿 샘플 이미지를 사용하지 않는다.
- `profile.png`, `profile2.png`, `screenshot.png`를 프로젝트 cover로 재사용하지 않는다.
- 외부 이미지를 출처와 권리 확인 없이 다운로드하지 않는다.
- AI 생성 이미지를 프로젝트 실제 화면처럼 사용하지 않는다.

cover와 gallery는 실제 Astro 화면을 촬영하고 Owner가 내용·권리·alt를 승인한 뒤에만 frontmatter에 추가한다.
